# Article Analyst Agent 테스트 시나리오

본 문서는 `article_analyst_agent`가 `plan.md`에 명시된 의도대로 작동하는지 검증하기 위한 상세 시나리오입니다. 사용자는 각 단계별로 에이전트의 응답과 '상태 표시기(st.status)'를 통해 작동 여부를 확인할 수 있습니다.

## 시나리오 1: 단일 기사 URL 분석 (MVP 핵심)
- **입력**: `https://www.bbc.com/news/business-68700000` (또는 활성화된 뉴스 URL)
- **기대 작동 흐름**:
    1. **Classifier**: `url_analysis`로 분류.
    2. **Agent**: `article_parser_tool` 호출 결정.
    3. **Tools (Parser)**: 본문 텍스트 성공적 추출.
    4. **Validator**: 본문 품질 "PASS" 판정.
    5. **Analyst**: 심층 분석 수행.
    6. **Tools (Report)**: `reports/` 폴더에 Markdown 파일 생성.
- **체크포인트**: 
    - [ ] 화면에 분석 리포트가 렌더링되는가?
    - [ ] 논조(Tone), 편향성(Bias) 항목이 포함되어 있는가?
    - [ ] 리포트 다운로드 버튼이 활성화되는가?
    - 정상작동

## 시나리오 2: 키워드 기반 뉴스 검색 및 분석
- **입력**: `"최신 인공지능 트렌드 요약해줘"`
- **기대 작동 흐름**:
    1. **Classifier**: `search_and_analyze`로 분류.
    2. **Agent**: `web_search_tool` 호출 결정.
    3. **Tools (Search)**: Tavily를 통해 관련 뉴스 리스트 확보.
    4. **Agent**: 검색 결과 중 가장 적절한 기사 선택 후 `article_parser_tool` 호출 유도.
    5. **이후 과정**: 시나리오 1과 동일하게 본문 추출 및 분석 진행.
- **체크포인트**:
    - [ ] 관련 뉴스를 찾았다는 상태 메시지가 표시되는가? - 정상작동
    - [ ] 검색된 내용에 기반하여 새로운 리포트가 생성되는가?
    - 리포터는 생성되지 않으나 뉴스는 잘 가져옴 하지만 요약이나 분석에 대한 것이 아예없음

## 시나리오 3: 분석 결과에 대한 후속 질문 (Context Awareness)
- **입력**: (시나리오 1 완료 후) `"이 기사에서 언급된 핵심 리스크 3가지만 다시 정리해줘"`
- **기대 작동 흐름**:
    1. **Classifier**: `followup_question`으로 분류.
    2. **Analyst**: 기존 대화 맥락과 리포트 내용을 바탕으로 답변 생성.
    3. **Agent**: 추가 도구 호출 없이 직접 답변 출력.
- **체크포인트**:
    - [ ] 이전 대화 내용을 기억하고 답변하는가? - 정상작동X
    - [ ] 답변이 리포트 요약보다 구체적인가?   - 정상작동X

## 시나리오 4: 유효하지 않은 입력 처리 (에러 핸들링)
- **입력**: `"안녕하세요?"` 또는 `"https://google.com"`
- **기대 작동 흐름**:
    1. **Classifier**: `general_chat` 또는 `url_analysis` 분류.
    2. **Validator (URL인 경우)**: 품질 "FAIL" 판정.
    3. **Agent**: 분석 불가 사유 안내 또는 일반 대화 수행.
- **체크포인트**:
    - [ ] 시스템이 중단되지 않고 적절한 안내 메시지를 출력하는가?  - 정상작동
    - [ ] `No AIMessage found`와 같은 기술적 에러가 발생하지 않는가?  - 정상작동

---

## 🛠️ 테스트 수행 방법
1. Streamlit 앱 실행: `streamlit run streamlit_app.py`
2. 좌측 사이드바의 **🔄 대화 초기화** 버튼을 눌러 상태를 리셋합니다.
3. 각 시나리오의 입력을 채팅창에 입력하고 **상태바(st.status)**의 변화를 관찰합니다.
4. 결과가 `reports/` 폴더에 실제로 생성되는지 확인합니다.
